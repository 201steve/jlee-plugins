# PRD 생성 스킬

## 개요

사용자의 요구사항을 받아 3개 에이전트 역할(BA → 기획자 → QA)을 순서대로 수행하여 완성된 PRD 마크다운 파일을 생성한다.

## 입력 처리

1. **파일 첨부가 있는 경우**: 첨부된 파일의 내용을 요구사항으로 사용
2. **파일 첨부가 없는 경우**: 사용자의 메시지 텍스트를 요구사항으로 사용

## 실행 절차

### Step 1 — 기능명 추출

요구사항에서 핵심 기능명을 추출한다. 파일명으로 사용할 수 있도록 영문 소문자 + 하이픈 형식으로 변환한다.
- 예: "소셜 로그인" → `social-login`
- 예: "장바구니 기능" → `cart`

### Step 2 — BA (Business Analyst) 역할

요구사항을 아래 8개 카테고리 기준으로 분석하여 PRD 초안을 작성한다. 명시되지 않은 항목은 `TBD`로 표기한다.

**분석 카테고리:**
1. **목적(Purpose)**: 해결하려는 핵심 문제, 지금 필요한 이유
2. **대상 사용자(Target Users)**: 사용자 역할, 권한 수준
3. **사용자 흐름(User Flow)**: 주요 사용 시나리오의 단계별 흐름
4. **비즈니스 규칙(Business Rules)**: 제약 조건, 유효성 검증 로직
5. **상태 전이(State Transitions)**: 주요 상태 전이
6. **시스템 연동(System Integration)**: 기존 모듈/API와의 연결 방식
7. **비기능 요구사항(NFR)**: 성능, 보안, 접근성
8. **범위 및 우선순위(Scope & Priority)**: MVP 범위, 후순위 항목

**초안 PRD 섹션 구성:**

```markdown
# PRD: {기능명}

**상태**: DRAFT  
**작성일**: {오늘 날짜}

---

## 1. 개요 (Overview)

### 목적
{해결하려는 문제와 필요성}

### 대상 사용자
{사용자 역할과 권한}

### 성공 지표
{측정 가능한 성공 기준, 없으면 TBD}

---

## 2. 사용자 스토리 (User Stories)

| ID | 역할 | 목표 | 우선순위 |
|----|------|------|----------|
| US-001 | {역할} | {목표} | P0/P1/P2 |

---

## 3. 기능 요구사항 (Functional Requirements)

### FR-001: {요구사항 제목}
- **설명**: {상세 설명}
- **비즈니스 규칙**: {제약 조건}
- **인수 기준(AC)**: {완료 판단 기준}

---

## 4. 테스트 시나리오 (Test Scenarios)

### TC-001: {시나리오 제목}
- **Given**: {사전 조건}
- **When**: {행동}
- **Then**: {기대 결과}

---

## 5. 미결 사항 (Open Questions)

| ID | 질문 | 상태 |
|----|------|------|
| OQ-001 | {TBD 항목에서 파생된 질문} | 미결 |
```

### Step 3 — 기획자 (Planner) 역할

BA가 작성한 초안을 아래 5개 차원으로 검토하고, PRD 내용을 직접 보완한다.

**검토 차원:**
1. **사용자 여정 완성도**: 모든 경로(정상/비정상)가 문서화됐는지, 진입점이 명확한지
2. **비즈니스 로직 명확성**: 규칙이 구체적인지, 엣지 케이스가 처리됐는지
3. **오류 및 엣지 케이스 UX**: 로딩/빈 상태/에러 메시지가 정의됐는지
4. **연동 일관성**: 기존 시스템 패턴과 맞는지
5. **범위 및 실현 가능성**: MVP 범위가 명확히 구분됐는지

**보완 방식:**
- 누락된 사용자 스토리 추가
- 모호한 기능 요구사항 구체화
- 빈 상태/에러 UX 시나리오 추가
- TBD 항목 중 유추 가능한 것은 내용 채우기
- 미결 사항에 기획 관점의 질문 추가

### Step 4 — QA 역할

기획자가 보완한 PRD를 아래 5개 차원으로 검토하고, 테스트 시나리오를 보강한다.

**검토 차원:**
1. **테스트 가능성**: 인수 기준이 측정 가능한지
2. **엣지 케이스 및 경계 조건**: 입력 제한, null 값, 동시 접근
3. **상태 전이**: 모든 전이가 문서화됐는지, 유효하지 않은 전이 처리 여부
4. **오류 처리 완성도**: 에러 코드 매핑, 재시도 전략
5. **인수 기준 커버리지**: 부정 케이스(Negative case) 포함 여부

**보완 방식:**
- 누락된 테스트 시나리오 추가 (Given/When/Then 형식)
- 경계값 테스트 케이스 추가
- 부정 케이스(잘못된 입력, 권한 없음, 타임아웃 등) 추가
- 인수 기준이 모호한 기능 요구사항에 AC 보완
- 미결 사항에 QA 관점의 질문 추가
- 이슈는 심각도 표시: **[Critical]** / **[Major]** / **[Minor]**로 인수 기준에 주석 추가

### Step 5 — 파일 저장

완성된 PRD를 작업 폴더에 저장한다.

- **파일명**: `PRD-{기능명}.md` (기능명은 Step 1에서 추출한 영문 소문자 + 하이픈)
- **저장 위치**: 사용자가 선택한 작업 폴더

## 실행 규칙

- 3개 역할을 사용자 확인 없이 자동으로 순서대로 수행한다
- 각 단계는 이전 단계 결과물을 직접 수정/보완한다 (별도 섹션 추가 X)
- 명시되지 않은 항목은 TBD로 표기하고 미결 사항에 질문으로 등록한다
- 유추 가능한 항목은 합리적인 가정으로 채우되, 가정임을 인수 기준에 명시한다
- 기술 용어(API, endpoint, CRUD, JWT, OAuth 등)는 영문 그대로 사용한다
- 작업 완료 후 저장된 파일 경로와 함께 PRD 완성을 알린다
